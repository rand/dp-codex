# Documentation Style Guide

This project uses a consistent voice: precise, practical, lightly playful, never vague.

## Tone

1. Write like an experienced engineer explaining the fastest safe path.
2. Use dry humor sparingly.
3. Never let style obscure the command or decision.

## Structure

1. Lead with intent and audience.
2. Keep steps numbered when sequence matters.
3. Show commands in executable form.
4. Include expected outcomes and failure recovery.

## Quality Rules

1. No placeholder sections.
2. No aspirational instructions that are not empirically validated.
3. Keep cross-links current when files move.
4. Update docs in the same change as behavior changes.

## Voice Examples

Preferred:

1. "Run this check before commit to catch drift early."
2. "If this fails, the message should tell you exactly what to fix."

Avoid:

1. "Maybe do this if needed."
2. "The tool should probably work."

Professional confidence beats motivational theater every time.
